Project Walkthrough
Compliant GCS Bucket Terraform Module

A reusable Terraform module for GCP Cloud Storage that hard-codes six NIST 800-171 controls into every bucket it provisions, with plan-time validation that rejects non-compliant configuration before a single resource reaches GCP.

terraform gcp nist 800-171 cloud storage · kms · cmek · iam
01 //

Problem Statement

Most organizations treat compliance as a checklist filled out after infrastructure is already built. Security settings get bolted on after the fact, documentation lags behind the actual state of the environment, and auditors end up reviewing a system that looks compliant on paper but was never engineered to be.

This module inverts that model entirely. Instead of checking buckets for compliance after deployment, the controls are the infrastructure. Every Cloud Storage bucket provisioned by this module is compliant by construction — a consumer cannot misconfigure their way out of it, and the evidence is generated automatically at apply time.

02 //

Architecture Decision

Every design choice was made to eliminate the gap between what the policy says and what the infrastructure actually does.

// hard-coded controls
No Optional Security Settings
Encryption, public access prevention, uniform IAM, versioning, and retention are non-negotiable. A consumer can configure bucket names and retention floors — they cannot toggle off a control.
// plan-time validation
Fail Before Any API Call
Terraform validation blocks catch misconfiguration at plan time. A prod bucket with less than 365 days retention fails with a human-readable error before anything reaches GCP.
// attestation output
Machine-Readable Evidence
Post-apply, a structured output surfaces every enforced control with its value and mapped control ID. An auditor reads the Terraform state directly without a console login.
// separate kms keyring
Customer-Managed Keys (SC-12)
Each bucket gets its own KMS keyring. GCP holds no root key material and key lifecycle is fully under the customer's control with 90-day automatic rotation enforced at the module level.
// module architecture

Controls Enforced

Six NIST 800-171 controls are hard-coded into every bucket the module provisions. None of them are optional.

04 //

How It Works

The module enforces controls at three distinct layers, making it impossible to deploy a non-compliant bucket without Terraform explicitly refusing to proceed.

hard-coded resources
Controls Are the Infrastructure
Encryption, public access block, uniform IAM, and versioning are fixed resource attributes — not variables. A consumer calling the module cannot override them any more than they can rewrite the GCP API.
validation blocks
Plan-Time Rejection
Terraform validation blocks check the retention floor against the environment type before any API call is made. A prod deployment with less than 365 days retention fails at plan time with a clear, human-readable error — not a runtime failure after resources are partially created.
negative test consumer
Proof the Gate Fires
The consumers/negative-test/ directory contains a deliberately misconfigured consumer. The bucket is named should-never-exist because it never reaches GCP — the validation gate fires and Terraform exits before any resource is created. This is testable, repeatable proof that the enforcement works.
compliance_attestation
Automatic Evidence at Apply Time
After a successful apply, the compliance_attestation output surfaces every enforced control as machine-readable structured data — control ID, value, and enforcement method. An auditor can read the Terraform state directly without a console login or a manual evidence collection process.
05 //

Module Structure

The module is organized so the compliance logic, consumer examples, and test cases are all clearly separated and independently runnable.

tools/terraform/
  modules/compliant-gcs-bucket/ # the reusable module
    main.tf # bucket + KMS resources
    variables.tf # inputs with validation blocks
    outputs.tf # compliance_attestation output
  primitives/compliant-gcs/ # dev consumer example
    evidence/lab-2-4/ # generated attestation artifacts
  consumers/negative-test/ # proves the validation gate fires
// deploy the compliant bucket
# Initialize and apply
terraform init
terraform plan -var="environment=prod"
terraform apply

# View the compliance attestation output
terraform output compliance_attestation

# Run the negative test to prove the gate fires
cd consumers/negative-test
terraform plan # exits with validation error
06 //

The Strongest Angle

Most compliance tools check whether controls are in place after the fact. This module makes non-compliance impossible to deploy in the first place. That's the shift from compliance theater to compliance engineering.

// key insight
The audit evidence trail is built into the infrastructure itself. The compliance_attestation output is generated by the same process that creates the bucket — not assembled afterward by a human. There's no gap between what was deployed and what the evidence says was deployed.
