Project Walkthrough
Compliant S3 Primitive — NIST 800-53 as Terraform

A Terraform module that enforces five NIST 800-53 controls directly into every AWS S3 bucket it provisions. After apply, terraform show -json produces machine-readable compliance evidence an auditor can traverse without a console login or a screenshot.

terraform aws s3 nist 800-53 kms · iac-as-evidence · hipaa
01 //

Problem Statement

Every auditor eventually asks the same question: "Show me that your S3 buckets are encrypted, access-controlled, and logging." The typical answer is a console screenshot taken that morning — a point-in-time artifact with no chain of custody, no version history, and no way to prove the configuration was in place before the audit started.

This primitive inverts that. The Terraform state is the evidence. Because the controls are defined in code and applied through a plan-and-apply cycle, the configuration is version-controlled, repeatable, and machine-readable. An auditor can read the state file directly — no console access required, no screenshots involved.

02 //

Architecture Decision

The core insight is that Terraform both enforces the control and generates the audit artifact in a single step. There is no gap between what was deployed and what the evidence says.

// iac as evidence
One Step: Deploy and Attest
terraform apply enforces the controls. terraform show -json produces the evidence. Both happen in the same pipeline run — the state file is the audit artifact, not a separate document someone has to maintain.
// kms encryption
Customer-Managed Keys (SC-28)
AES-256 encryption via KMS with customer-managed keys. The key policy, rotation schedule, and ARN are all surfaced in the state — giving auditors direct evidence of encryption at rest without a console login.
// access control
Public Access Block (AC-3)
All four public access block settings are enforced at the resource level. No bucket provisioned by this module can be made public — the block is a fixed attribute, not a variable.
// audit logging
Access Logging (AU-3, AU-6)
Server access logging is enabled and pointed at a dedicated log bucket. Every object-level operation is captured. The log destination is enforced in the module — consumers can't disable it.
// module architecture

Controls Enforced

Five NIST 800-53 controls are hard-coded into every bucket this module provisions. The same controls also satisfy HIPAA Technical Safeguard requirements for ePHI storage.

The Evidence Artifact

The key insight of this primitive is that Terraform both enforces the control and generates the audit artifact in one step. Running terraform show -json after apply produces structured, machine-readable output that maps directly to each control. No console screenshots, no manual evidence collection, no gap between what was deployed and what the evidence says.

// key insight
An auditor reading the Terraform state can verify SC-28 is satisfied by checking the server_side_encryption_configuration block. AC-3 is verified by the public_access_block resource. AU-3 is verified by the logging block. Every control has a corresponding state field — no interpretation required.
// generate machine-readable evidence
# After terraform apply, run:
terraform show -json | jq '.' > compliance-evidence.json

05 //

How It Works

main.tf
Hard-Coded Control Attributes
Encryption, public access block, versioning, ownership controls, and logging are defined as fixed resource attributes. A consumer calling the module passes a bucket name and tags — they cannot pass parameters that override the security settings.
kms key resource
Customer-Managed Encryption
A dedicated KMS key is created alongside the bucket. The key ARN is passed directly into the S3 encryption configuration. AWS holds no root key material under the customer's account — SC-28 and SC-13 are satisfied at the infrastructure level.
log bucket resource
Dedicated Audit Log Destination
A separate S3 bucket is created to receive server access logs. The logging configuration on the primary bucket points to this destination. Because both resources are managed by the same module, the log destination cannot be removed without destroying the primary bucket first.
outputs.tf
Structured Evidence Output
The module exposes outputs for bucket ARN, KMS key ARN, and log bucket ARN — all the references an auditor needs to verify the control chain. Combined with terraform show -json, these outputs make the entire configuration auditor-traversable from the state file alone.

Module Structure

tools/terraform/primitives/compliant-s3/
  main.tf # s3 bucket + kms key + log bucket resources
  variables.tf # bucket name, tags, environment inputs
  outputs.tf # bucket arn, kms arn, log bucket arn
  versions.tf # provider version constraints
  evidence/ # generated compliance-evidence.json artifacts
// deploy the primitive
# Initialize providers
terraform init

# Review what will be created
terraform plan -var="bucket_name=my-compliant-bucket"

# Apply and generate evidence in one step
terraform apply -auto-approve
terraform show -json > evidence/compliance-evidence.json

Summary

The S3 primitive demonstrates a repeatable pattern for compliance-as-code: define controls in infrastructure, deploy them through a version-controlled pipeline, and surface the evidence from the state file. The same pattern scales to any cloud resource — this bucket is the first primitive in a broader library.
