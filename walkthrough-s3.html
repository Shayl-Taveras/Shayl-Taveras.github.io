Compliant S3 Primitive

Five NIST 800-53 controls expressed as Terraform. One JSON file replaces every screenshot an auditor would otherwise request.

terraform nist 800-53 aws s3 kms opa iac-as-evidence hipaa
01 //

The Problem

Most compliance evidence is still screenshots and spreadsheets: point-in-time captures that are manual, easy to fake, and impossible to diff. When an auditor asks "prove S3 encryption is on," the default response is a console screenshot with a timestamp. It tells you the state at that moment. It says nothing about tomorrow.

The goal: encode the control requirement directly into the infrastructure so the enforcement and the evidence are the same artifact. The Terraform is the policy. The state file is the proof.

The Architecture Decision

The choice to use Terraform over a Python script or console configuration is a GRC-informed infrastructure decision. Terraform state is machine-readable, versionable, and diffable. A console config is a fact; Terraform state is a verifiable, repeatable record of that fact.

enforcement
IaC as Policy
Controls aren't documented alongside the code. They are the code. SC-28 isn't a note in a README; it's an aws_s3_bucket_server_side_encryption_configuration block that fails to apply if removed.
evidence
State as Artifact
terraform show -json produces a structured state file where an auditor can trace SC-28 to the SSE config, AC-3 to four explicit denies, and AU-3 to the logging target, all in a single JSON traversal.
03 //

The Five Controls

Each control maps to a specific Terraform resource. The comment in the code cites the control ID so any engineer reading a failed PR knows exactly what compliance requirement they broke.

SC-28
Protection of Information at Rest
AES-256 server-side encryption enforced via aws_s3_bucket_server_side_encryption_configuration. All four flags on the public access block must be true. Three is not enough. AWS treats them as independent doors.
Terraform: aws_s3_bucket_server_side_encryption_configuration.primary
AC-3
Access Enforcement
Explicit deny on every public access vector via aws_s3_bucket_public_access_block. All four flags set to true: block_public_acls, block_public_policy, ignore_public_acls, restrict_public_buckets.
Terraform: aws_s3_bucket_public_access_block.primary
CM-6
Configuration Settings
Two mechanisms. Versioning preserves prior object states for recovery and audit. Compliance tags (Project, Environment, ManagedBy, ComplianceScope) are set on the provider with default_tags, making them non-optional on every taggable resource.
Terraform: aws_s3_bucket_versioning.primary · provider default_tags
AU-3
Content of Audit Records
S3 server access logging routes to a dedicated log bucket. Logs in the same bucket could be deleted or overwritten alongside the data they record. Separation makes the audit trail harder to tamper with.
Terraform: aws_s3_bucket_logging.primary · aws_s3_bucket.log
AU-6
Audit Record Review, Analysis, and Reporting
The log bucket is independently hardened. Its own encryption configuration and public access block ensure the audit trail itself meets the same baseline as the data bucket. An unsecured log bucket is an audit finding waiting to happen.
Terraform: aws_s3_bucket_server_side_encryption_configuration.log · aws_s3_bucket_public_access_block.log
04 //

Key Technical Decisions

Every non-obvious choice in the module has a specific reason rooted in either compliance defensibility or operational reliability, not preference.

DecisionReasoning
default_tags on provider Compliance tags are non-optional. Setting them once on the provider eliminates the risk of any resource being deployed untagged, a single source of truth rather than a per-resource tax.
Separate log bucket Logs in the same bucket as the data they record can be deleted or overwritten in the same operation. Separation makes the audit trail harder to tamper with and is what most framework auditors expect.
random_id suffix S3 bucket names are globally unique across all AWS accounts. A random suffix prevents name collisions on first deploy without manual coordination across environments.
one() expression Terraform stores SSE rules as a set, not an indexable list. one() safely extracts the single element and fails loudly if configuration drift ever adds a second rule, catching misconfiguration at output time, not at audit time.
terraform show -json Machine-readable state is more auditable than screenshots. It can be parsed, diffed, and verified programmatically by OPA in later labs. The JSON is the evidence artifact, not a byproduct of deployment.
Variable validation blocks Catches bad inputs (wrong environment name, invalid project name) at plan time with clear error messages rather than cryptic AWS API failures at apply time. Fail early, fail clearly.
05 //

The Evidence Output

After a successful apply, two commands capture machine-readable compliance evidence. No console login. No screenshots. The state file is a structured JSON record an auditor or OPA policy to traverse programmatically.

// capture evidence — runs after every apply
# Capture the plan artifact
terraform show -json tfplan > evidence/plan.json

# Capture live state — this IS the compliance attestation
terraform show -json > evidence/state.json
// what an auditor finds in state.json
# SC-28 — encryption at rest
"sse_algorithm": "AES256"

# AC-3 — all four public access denies
"block_public_acls": true,
"block_public_policy": true,
"ignore_public_acls": true,
"restrict_public_buckets": true

# CM-6 — versioning and tags
"versioning_configuration": [{ "status": "Enabled" }]
"tags": { "ComplianceScope": "cge-p-lab", "ManagedBy": "terraform" }

# AU-3 — logging target confirmed
"logging": [{ "target_bucket": "cgep-lab-dev-logs-XXXXXXXX" }]
// key insight
terraform show -json output IS the audit artifact. The same tool that deploys the infrastructure produces a structured record of its compliance state. The enforcement step and the evidence step are a single command, not two separate workflows that can drift apart.
